# Раздел «Заказы» в админке Strapi (экран сборки и фискализации)

Дата: 2026-07-20. Статус: дизайн согласован (заказчик исполнителя: «полный раздел, аккуратно и качественно»).
Решение проджекта: «Делаем свою на Strapi. Lovable-админку не юзать вообще».

## 1. Зачем

По ТЗ маркированные товары продаются онлайн наравне с остальными (стр. 363–365), код
«Честного знака» сканируется сотрудником при сборке до фискализации (стр. 366–367), приёмка
Этапа 3 требует «заказ с маркированным товаром фискализируется с кодом маркировки»
(стр. 445–446). Из-за согласованных отклонений (Путь B: Эвотор read-only; фискализация через
ЮKassa, а не «Цифровую кассу») этот сценарий не может жить на кассе Эвотора — ему нужно
рабочее место в нашей админке. Заодно закрывается сценарий выдачи самовывоза «кассир видит
заказ в админке» (согласованная замена пречека) и задача «реальный раздел Заказы».

Бэкенд полностью готов и оттестирован (270 юнитов; сквозной прогон оплаты на проде пройден
20.07). Работа — только страница в Strapi + серверный мостик к готовому API.

## 2. Архитектура

Локальный плагин Strapi `orders` (apps/strapi/src/plugins/orders), Strapi v5.

```
Браузер (страница «Заказы» в админке Strapi; пользователь уже залогинен в Strapi)
  → серверные руты плагина  /orders/…  (policy: только аутентифицированный админ Strapi)
    → прокси-клиент → бэкенд http://127.0.0.1:3002/api/v1/admin/*  (готов)
```

Заказы НЕ хранятся в Strapi и НЕ мигрируются в него. Никакой бизнес-логики в плагине:
все инварианты (граф статусов, гейт «не выдать маркированный без чека», гейт отмены
фискализированного, дедуп кодов, лимит 80 позиций, идемпотентность фискализации) уже
живут в бэкенде — плагин только отображает данные и переводит ошибки в понятный текст.

### Мостик авторизации (без второго пароля для людей)

- Вход людей — обычный логин Strapi (существующие аккаунты: заказчица-контент-менеджер,
  исполнитель-админ). Новых учёток и паролей не заводим.
- Прокси-клиент на сервере Strapi сам логинится в бэкенд: `POST /api/v1/admin/login` с
  паролем из env Strapi (`ORDERS_ADMIN_PASSWORD` = ADMIN_PASSWORD бэкенда, уже есть в
  прод-.env бэкенда; в Strapi добавляется своя переменная), кэширует полученный токен в
  памяти, при 401 повторяет логин один раз и перезапрашивает. Таймаут запросов к бэкенду 10 с.
- env плагина: `ORDERS_API_URL` (default `http://127.0.0.1:3002/api/v1`),
  `ORDERS_ADMIN_PASSWORD` (обязателен на проде; без него раздел показывает понятную
  ошибку конфигурации, Strapi не падает).

### Используемые ручки бэкенда (все существуют)

- `POST /admin/login` — сервисный логин прокси.
- `GET /admin/orders?status=&deliveryMethod=&page=` — список.
- `GET /admin/orders/:id` — карточка (items c isMarked/markCodes, fiscalizationRequired,
  fiscalizationInProgress, fiscalReceiptId).
- `PATCH /admin/orders/:id/items/:itemId/mark-codes` — сохранить коды позиции.
- `POST /admin/orders/:id/fiscalize` — выбить отложенный чек (зачёт предоплаты).
- `PATCH /admin/orders/:id/status` — смена статуса (с причиной для отмены).

## 3. UI (дизайн-система Strapi, всё по-русски)

### 3.1 Список заказов

- Пункт меню «Заказы» в левой навигации админки.
- Таблица: № заказа, дата, клиент (имя, телефон), сумма, получение (Ленинградская /
  Титова / курьер / Россия), оплата (онлайн-оплачен / при получении), статус-бейдж
  (цвета по статусам), индикатор «маркировка» (требуется фискализация / чек выбит).
- Фильтры: статус, способ получения. Пагинация серверная. Автообновление — поллинг 30 с
  (без вечных спиннеров: тихое фоновое обновление).
- Пустые состояния и ошибки сети — человеческим текстом, с кнопкой «Повторить».

### 3.2 Карточка заказа

- Блоки: клиент и получение; оплата (способ, статус, номер платежа); комментарий;
  позиции (название, кол-во, цена, итог; метка «маркированный»); итоги (скидка, доставка).
- Блок сборки маркированного — только если заказ оплачен онлайн и содержит маркированные
  позиции: под каждую ЕДИНИЦУ товара своё поле кода (qty=2 → 2 поля), автофокус и
  автопереход на следующее поле после ввода (сканер = клавиатурный ввод + Enter),
  счётчик «введено N из M», сохранение кодов по позиции (PATCH) с индикатором успеха.
- Кнопка «Фискализировать»: активна только когда все коды всех маркированных позиций
  введены; подтверждение перед отправкой; после успеха — плашка «Чек выбит» с id чека;
  при fiscalizationInProgress — блокировка повторного нажатия.
- Смена статуса: показываются только допустимые переходы из текущего статуса
  (new/paid → собирается; собирается → готов к выдаче | передан в доставку;
  готов/передан → выполнен; отмена — до сборки, с обязательной причиной и подтверждением).
  Правила дублируют граф бэкенда только для видимости кнопок; истина — бэкенд.

### 3.3 Перевод ошибок бэкенда (обязательный словарь)

| Код бэкенда | Текст пользователю |
|---|---|
| ORDER_NOT_FISCALIZED | «Сначала отсканируйте коды и выбейте чек — потом выдача» |
| ORDER_FISCALIZED | «Чек уже выбит — отмена только после чека возврата в ЮKassa» |
| ORDER_TRANSITION_FORBIDDEN | «Действие недоступно для текущего статуса (обновите страницу)» |
| ошибки кодов маркировки (дубль, неполный набор, «кодов X, а единиц Y») | текст бэкенда как есть — он уже человекочитаемый |
| сеть/5xx | «Не удалось связаться с сервером заказов. Повторите» |
| ошибка конфигурации прокси | «Раздел не настроен: нет доступа к серверу заказов (ORDERS_ADMIN_PASSWORD)» |

## 4. Тестирование и приёмка

1. Юниты серверной части плагина (чистые модули): кэш токена и повтор при 401,
   маппинг ошибок, сборка query-параметров. Бэкенд-юниты уже есть (270).
2. Локально: сборка Strapi (`build` админки) зелёная; страница проверяется на локальном
   Strapi против прод-бэкенда только чтением (список/карточка), мутации — на проде на
   тестовом заказе.
3. Прод: деплой только Strapi → живой сквозной прогон приёмочного сценария:
   тестовый маркированный заказ (имя «ТЕСТ …») → оплата тестовой картой ЮKassa →
   в админке: скан (ввод) фейк-кодов → «Фискализировать» → чек зачёта в тестовой кассе →
   перевод «готов к выдаче» (гейт пропускает после чека) → «выполнен» → уборка тест-заказа.
4. Негативные проверки на проде: попытка выдачи ДО фискализации (ждём понятную ошибку),
   повторное нажатие «Фискализировать» (идемпотентность), отмена фискализированного (отказ).

## 5. Деплой

- Меняется только apps/strapi (+ удаление Lovable-мокапа `/admin` из apps/frontend —
  решение проджекта; отдельный маленький коммит и пересборка фронта).
- Сборка Strapi локально → артефакт на сервер → `pm2 restart altai-strapi`.
  Бэкенд не трогается. Откат: предыдущая папка Strapi сохраняется рядом.

## 6. Вне объёма (осознанно)

- Печать документов/чеков из админки.

## 7. Критерии «сделано»

- Заказчица под своим обычным логином Strapi видит «Заказы», открывает заказ,
  сканирует коды, жмёт «Фискализировать», меняет статусы — без единого технического слова.
- Живой прогон п.4.3 пройден на проде целиком, негативные проверки п.4.4 дают понятные тексты.
- Юниты плагина + существующие 270 юнитов бэкенда зелёные; сборки Strapi и фронта зелёные.
