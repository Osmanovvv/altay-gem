# По-магазинное наличие на витрине (недочёт #5 ТЗ / задача #37)

Дата: 2026-07-22. Статус: утверждено пользователем (вариант 1, точные числа).

## Проблема

Витрина показывает остаток, агрегированный по двум магазинам (ТЗ:266), а заказ
резервирует остаток из ОДНОГО магазина: самовывоз — выбранная точка, доставка —
магазин «основной» записи товара. Товар, лежащий только на Ленинградской, горит
«в наличии», покупатель выбирает самовывоз на Титова и получает отказ
`out_of_stock` на последнем шаге. Денег/порчи данных нет — но отказ на финале
это плохой UX и жалобы заказчице.

Вторая нечестность: буфер «последний экземпляр» (ТЗ п.8) на витрине вычитается
один раз ИЗ СУММЫ (Лен 3 + Тит 2 − 1 = «4»), а при заказе — по-магазинно
(максимум 2 из одной точки). Витринное число недостижимо даже теоретически.

## Решение (утверждено: «вариант 1»)

Агрегат НЕ убираем (ТЗ:266 прямо требует агрегацию) — дополняем его честной
разбивкой и ранней проверкой.

### A. Честный агрегат

`availableQty` карточки = **сумма по-магазинных буферизованных** значений.
По каждому магазину: `applyStockBuffer(floor(порции(физостаток − активные
резервы)), buffer)`, затем сумма. Для весовых порции считаются floor'ом
ПО КАЖДОМУ магазину до буфера (0.95 кг в каждой точке при порции 1 кг = 0,
а не 1 — заказать эту порцию из одной точки всё равно нельзя).
`inStock = availableQty > 0`. Это ровно та математика, по которой create()
проверяет заказ, — витрина больше не обещает недостижимое.

### B. Разбивка по точкам в API карточки

Карточка (и деталь, она наследует карточку) получает поле:

```ts
pickupAvailability: Array<{
  point: 'pickup_leningradskaya' | 'pickup_titova';
  availableQty: number; // буферизованное, в единицах продажи (шт/порции)
}>
```

Считается в `buildAll()` из тех же данных (реплика + резервы + буфер),
кэшируется вместе с каталогом (TTL 60с, инвалидация как сейчас). Магазины,
не являющиеся точками самовывоза, в разбивку не попадают (агрегат — по всем).

### C. Общий модуль точек самовывоза

`src/orders/pickup-points.ts`: `PICKUP_STORE_HINT` (метод → подстрока адреса)
и `resolvePickupStores(stores)` → `[{point, storeId}]`. Используют оба:
`orders.service.resolveTargetStore` (рефактор, поведение то же, включая
`PICKUP_POINT_UNKNOWN`) и `catalog.service.buildAll` (одна тонкая выборка
`evotor_stores`, кэшируется с каталогом).

### D. Общий stock-check хелпер

Из цикла транзакции `create()` извлекается чистая функция
`orderableQty({physicalQty, reservedQty, measure, portionMassG, buffer})` →
доступно к заказу в единицах продажи. Используют:
- `create()` — как сейчас, внутри транзакции с FOR UPDATE (поведение не меняется);
- `quoteDelivery()` — новая предпроверка обычным SELECT (без блокировок);
- `catalog.buildAll()` — та же математика для витрины (A и B).

Одна функция — один источник правды, расхождение витрины/quote/create исключено.

### E. Ранняя проверка на чекауте (серверная)

`quoteDelivery()` дополнительно резолвит целевой магазин выбранного способа
(та же логика, что create: самовывоз → точка, доставка → магазин записи товара)
и проверяет каждую позицию корзины. Ответ дополняется полем:

```ts
stockProblems?: Array<{
  id: string;            // slug позиции
  availableQty: number;  // доступно в целевом магазине
  otherPickup?: { point: PickupPoint; availableQty: number };
  // otherPickup только для самовывоза и только если другая точка
  // покрывает запрошенное количество
}>
```

Ошибкой это НЕ является (HTTP 200, стоимость доставки считается как раньше) —
поле мягкое, фронт решает, что показать. Существующие поля ответа не меняются.

Фронт (checkout, шаг 1): при непустом `stockProblems` под способами доставки —
предупреждающий блок по позициям: «Сыр Граф: в этом пункте только 1 шт
(в корзине 3). Есть на Ленинградской 75/2 — 5 шт», переход на шаг 2
блокируется (как e-mail-гейт: ошибка + toast). Смена точки/количества
перезапрашивает quote (уже происходит) и снимает блок.

Авторитетный финальный гейт при создании заказа НЕ меняется.

### F. Страница товара

Под «В наличии: N» — блок «Самовывоз:» со строками по точкам:
«✓ Ленинградская 75/2 — 5 шт» / «✗ Титова 32 — нет в наличии»
(адреса — константа фронта, та же что в чекауте). Точные числа утверждены
пользователем. Для весовых — в порциях, как агрегат.

## Не входит (осознанно)

- Дробление заказа по двум магазинам (ТЗ не описывает, чек один).
- Смена правила «склад по умолчанию» для доставки (магазин записи товара);
  теперь оно хотя бы честно отбивается заранее на чекауте.
- Разбивка на карточках СПИСКА каталога (только страница товара + чекаут).
- Корзина (степпер капится новым честным агрегатом автоматически).

## Тесты

- Юнит: `orderableQty` (штучные/весовые/резервы/буфер/нули), агрегат=Σ точек,
  `resolvePickupStores`, quote со stockProblems (нужная/другая точка,
  доставка, otherPickup-условие), create() не изменил поведение.
- Обновить существующие ожидания (агрегат стал консервативнее — намеренно).
- E2E (headless playwright): деталь товара показывает разбивку; чекаут
  с «неправильной» точкой показывает предупреждение и не пускает дальше,
  с «правильной» — пускает; гейты create() не тронуты.
